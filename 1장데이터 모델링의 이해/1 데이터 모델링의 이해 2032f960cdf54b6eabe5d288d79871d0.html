1.데이터 모델링의 이해

  • 데이터 모델링이 필요한 주요 이유
    • 업무정보를 구성하는 정보들을 표기법으로 정보시스템 구축의 대상이 되는 업무를 분석
    • 분석된 모델을 가지고 실제 데이터베이스를 생성하여 개발 및 데이터관리에 사용하는것
      • 데이터 모델링은 데이터베이스만 구축하는 것이 아닌 데이터모델링 자체로서 업무를 설명하는 부분에도 중요함
  • 데이터 모델링을 할때 유의할 사항
    • 중복 되면 안된다.
    • 비유연성
      • 수시로 변경되면 유지보수 어려움
    • 비일관성
      • 데이터는 항상같아야한다.(상호 연관관계가 명확해야한다.)
  • 데이터 스키마구조
    • 외부스키마
      • 외부사용자
    • 개념스키마
      • 사용자 모든관점 통합
    • 내부스키마
      • 내부 데이터베이스
  • 엔터티
    • 엔터티는 2개이상의 속성과 2개이상의 인스턴스가 있어야 자격이됨
    • 엔터티
      • 테이블
    • 속성
      • 컬럼 (이름, 주소)
    • 속성값
      • 한셀의 값 (이한규,신림동 1개 값)
    • 인스턴스
      • 로우 (이한규, 신림동, 26) 한줄
      • 아래로 따지자면 한로우로 2, 악순이, 1989 하나의 줄을 말함
      • 연관관계
  • 엔터티의 특징
    • 업무에서 필요한 정보
    • 유일한 식별자에 의해 식별가능
    • 두개이상 인스턴스 집합
    • 업무 프로세스에 이용
    • 반드시 속성이 필요
    • 엔터티는 한개이상의 관계필요
  • 속성의 특징
    • 기본
      • 원래값
      • 일반
    • 설계
      • 업무 규칙으로 만들어진 값
      • 분류
    • 파생
      • 속성값을 게산하여 저장할수 있도록 만든 파생속성
      • 계산
  • 도메인
    • 정수의 10자리
    • 문자의 30자리
    • 속성의 범위설정
  • 속성의 명칭부여
    • 애매모호 금지
    • 복합명사 사용
    • 유일성(테이블마다 속성명 같으면 안됨)
    • 이래야 반정규화, 통합때 혼란 방지 가능
  • 데이터 모델링 관계
    • 존재에 의한 관계
      • 실선
    • 행위에 의한 관계
      • 점선
    • UML
      • 연관관계, 의존관계 구분 X
  • 관계 3가지 개념
    • 관계명
    • 관계차수
    • 선택성
  • 식별자 종류
    • 기본키 중간선 위에 있는것만 따지는것임 ㅇㅋ??

    • 주식별자 VS 보조식별자
      • 엔터티내 대표성을 구분
    • 내부식별자 VS 외부식별자
      • 엔터티내 스스로 생성되었는지 구분
    • 단일식별자 VS 복합식별자
      • 단일 속성인지 유무
    • 본질식별자 VS 인조식별자
      • 원래 있던 의미
      • 일련번호로 분류하여 새롭게 만든 식별자
  • 식별자 선정
    • 명칭, 내역, 이름은 중복성이 생길수 있어 안됨
  • 주식별자의 특징
    • 유일성
      • 인스턴스를 유일하게 구분함
      • (사람이란 인스턴스에 주민번호같이 구분)
    • 최소성
      • 주식별자의 속성의 수는 최소한으로
      • (복합으로 많으면 성능저하)
    • 불변성
      • 주식별자가 지정되면 변하지않아야함
      • (주민번호 안변함)
    • 존재성
      • 주식별자가 지정되면 NULL 안됨
  • 식별자와 비식별자 관계
    • 기본키와 외래키 같은느낌임
      • 기본키
        • 유일한 값
      • 외래키
        • 참조관계의 집합
        • 외래키의 식별자 (주키 → 주키AND외래키)
          • 주식별자나 보조식별자만 가능
        • 외래키의 비식별자 (주키 → 외래키)
          • 선택적인 관계
  • 성능 데이터 모델링
    • 분석설계 단계에서 데이터베이스 처리 성능을 향상 방법
      • 테스트 환경을 구현하고 그곳에 트랜젝션을 발생시켜 실제 성능을 테스트 해보아야한다.
      • 데이터모델의 구조를 변경하면서 어떠한 구조가 해당 사이트에 성능상 가장 적절한 구조인지 검토
      • 성능과 sql 튜닝은 무관함
  • 성능을 고려한 데이터 모델링의 순서
    1. 데이터 모델링할 때 정규화를 정확하게 수행
    1. 데이터베이스 용량산정 수행
    1. 데이터베이스에 발생되는 트랜잭션의 유형을 파악
    1. 용량과 트랜잭션의 유형에 따라 반정규화 수행
    1. 이력모델 조정, PK.FK 조정, 슈퍼타입, 서브타입 조정등 ㅜ행
    1. 성능관점에서 데이터모델 검증
  • 1차정규화 대상
    • 컬럼의 반복적인 형태로 원자성을 위배한 1차 정규화 대상
    • 이와같은경우 입력, 수정,삭제 성능저하 됨
    • 1차 정규화 수행후 인덱스 적용
  • 반정규화
    • 컬럼을 자주 이용하는 집중화된 컬럼이 있을때 컬럼들을 별도로 모으는것은 테이블추가 부분테이블 추가 이다
  • SQL문에서 조회를 빠르게 수행하려면 테이블을 전부 통합하여 조회한다.
  • 전체가 아닌 최근 변경된 값만 골라서 조회의 겨우 성능이 저하되서 나온다
  • 컬럼이 많은 테이블
    • 디스크I/O가 대량으로 발생하여 성능저하
    • 개선
      • 트랜잭션이 접근하는 칼럼 유형을 분석해서 빈도수 높은 칼럼과 낮은 칼럼을 1대1로 테이블 분리
      • 데이터가 채워진 칼럼을 앞으로 NULL이 분포된것은 뒤로
        • 하지만 NULL값이 갑자기 채워진다면 로우체인발생
  • 파티셔닝
    • 데이터 엑세스 향상
    • 논리적으로 하나의 테이블
    • 물리적으로 여러개의 테이블로 분리한 기법
  • 트랜잭션
    • 항상 전체를 통합형 분석처리
    • 슈퍼-서브타입이 하나의 테이블로 통합되어있으면 테이블에 연결된 데이터만 읽어 다른 형식에 비해 성능이 우수

  • 테이블 데이터 조회
    • ‘=’로 들어온 조건에 해당하는 칼럼이 인덱스의 가장 앞쪽 위치하면 효율이 높음
      • 1순위 : ‘=’
      • 2순위 : ‘BETWEEN’ OR ‘≤’
  • 외래키도 인덱스가 필요함 ㅇㅋ?
